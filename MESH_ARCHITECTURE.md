# Частичная P2P mesh-сеть с автоперестраиванием маршрутов

## Цель

`server.py` выполняет только роль control plane: аутентифицирует узлы,
хранит их состояние и выдаёт конфигурацию. Пакеты пользовательских сервисов
через него не проходят. Передача данных идёт по UDP между соседними P2P-узлами.

Сеть должна давать путь между узлами за symmetric NAT через доступные cone-узлы:

```text
symmetric A -> cone superpeer 1 -> cone superpeer 2 -> symmetric B
```

Каждый узел имеет постоянный `node_id` и пару ключей. Внешний UDP endpoint и
NAT-тип являются временными свойствами сессии, а не идентификатором узла.

## Двухуровневая топология

### Уровень 1 — backbone из cone-суперпиров

Суперпир — узел с NAT типа `cone`, который согласился пересылать транзитный
трафик. Он поддерживает устойчивые P2P-каналы с другими суперпирами.

`server.py` выбирает кандидатов по следующим метрикам:

- длительность непрерывной сессии;
- доступная пропускная способность и текущая загрузка;
- RTT/ping до других суперпиров;
- успешность keepalive и процент потерь;
- явно разрешённая владельцем роль relay.

Полная связность нужна только для небольшого backbone. Для большего числа
суперпиров каждый держит `5..7` соседей с наименьшей стоимостью пути, плюс
один резервный сосед из другой части графа. Это даёт избыточность без полного
графа.

### Уровень 2 — обычные узлы

Обычный узел, включая symmetric NAT, держит активные каналы с `2..3`
суперпирами. Их выбирает `server.py`, учитывая RTT, загрузку и независимость
маршрутов: по возможности суперпиры не должны находиться за одним домашним
маршрутизатором или в одной точке отказа.

Обычный узел не обязан принимать транзитный трафик. Его трафик к другому узлу
всегда сначала уходит к одному из своих суперпиров.

```text
                    +-------------------+
                    | server.py          |
                    | control plane only |
                    +---------+---------+
                              |
          +-------------------+-------------------+
          |                   |                   |
     cone SP-1 <---------> cone SP-2 <---------> cone SP-3
        /     \                 |                 /     \
   sym A     cone B            sym C          sym D     cone E
```

## Построение и хранение маршрутов

Каждый суперпир знает граф backbone и локально запускает Dijkstra по стоимости
каналов. Это не минимальное остовное дерево: одновременно сохраняются
несколько путей, поэтому отказ одного канала не разрушает сеть.

Стоимость ребра:

```text
cost = RTT_ms + loss_penalty + load_penalty + instability_penalty
```

Для каждого назначения суперпир хранит как минимум:

```text
destination_node_id -> [next_hop_1, next_hop_2, backup_next_hop]
```

Для обычного узла достаточно таблицы опубликованных сервисов и списка его
суперпиров. Он отправляет пакет выбранному суперпиру; тот выбирает следующий
hop по своей таблице.

`server.py` распространяет первоначальный снимок topology и список сервисов.
Суперпиры могут уточнять стоимости каналов сами по `PING/PONG`, не обращаясь к
серверу для каждого изменения RTT.

## Автоперестроение

1. Каждый канал посылает keepalive и измеряет RTT/потери.
2. После нескольких пропусков подряд канал помечается `suspect`, затем `down`.
3. Суперпир немедленно переключает новые пакеты на резервный `next_hop`.
4. Он отправляет соседям `LINK_DOWN`/`LINK_UP` с номером версии topology.
5. Суперпиры пересчитывают маршруты; `server.py` получает агрегированное
   состояние и при необходимости назначает обычному узлу новый суперпир.
6. При восстановлении канала он возвращается в граф только после серии
   успешных проверок, чтобы избежать флаппинга.

Для защиты от частых переключений нужны hysteresis и hold-down timer:
например, не менять основной маршрут, если новый лучше менее чем на 15%, и не
возвращать упавший канал в основной маршрут в течение 30 секунд стабильной
работы.

## Минимальный UDP overlay-протокол

Каждый пакет должен нести маршрутизируемый, но не содержащий полезных данных
заголовок:

```text
version | type | packet_id | source_node_id | destination_node_id | ttl | payload
```

Рекомендуемые типы:

- `HELLO`, `AUTH`, `SESSION_CONFIRM` — установление защищённого P2P-канала;
- `KEEPALIVE`, `PING`, `PONG` — поддержка NAT-мэппинга и метрики;
- `DATA` — зашифрованный payload сервиса;
- `ROUTE_UPDATE`, `LINK_UP`, `LINK_DOWN` — состояние backbone;
- `SERVICE_ANNOUNCE`, `SERVICE_QUERY` — каталог сервисов;
- `ERROR_NO_ROUTE`, `ERROR_TTL` — диагностические ответы.

`ttl` уменьшается на каждом суперпире (например, начальное значение 8).
Узел хранит небольшой кэш последних `packet_id` от каждого источника, чтобы
отбрасывать дубликаты при наличии резервных путей и не создавать петли.

## Безопасность

- `node_id` должен быть производным от публичного ключа, а не случайным ID
  текущего запуска;
- каждый control-plane и P2P-запрос подписывается или аутентифицируется;
- payload шифруется end-to-end между исходным и конечным узлом;
- суперпир видит только маршрутный заголовок и не может читать данные сервиса;
- relay ограничивает TTL, скорость, размер очередей и число транзитных сессий;
- правила доступа к сервису проверяются у конечного узла по публичному ключу
  отправителя, а не по IP-адресу.

## Поэтапное внедрение

1. Ввести постоянную идентичность узла и регистрацию в `server.py`.
2. Сделать один защищённый P2P-канал и передачу `DATA` между двумя соседями.
3. Добавить режим cone-суперпира: таблицу соседей, relay и TTL.
4. Добавить подключение обычного узла к двум суперпирам.
5. Реализовать статические маршруты, выдаваемые `server.py`.
6. Добавить измерение стоимости линков, резервный next hop и локальный Dijkstra.
7. Добавить публикацию домашних сервисов и end-to-end шифрование payload.

Полное сканирование портов следует оставить только как временную попытку
установить прямой UDP-канал. Для постоянных соседских соединений предпочтительны
долгоживущие каналы с keepalive и контролируемое число суперпиров.
